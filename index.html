Permission Explorer

How camera and microphone permissions actually work — from hardware to tap, bottom to top.

L0 Hardware — Physical Devices

Your iPhone has physical camera sensors and microphones. They sit idle until an app or website requests access. iOS routes all audio/video through the hardware bus managed by the kernel.

[Front Camera] ────┐ ├─→ ISP (Image Signal Processor) ─→ Frame buffer [Rear Camera] ────┘
[Microphone] ────→ Audio Codec (ADC) ─→ PCM samples
L1 iOS — OS-Level Privacy Gate

iOS enforces a mandatory privacy framework. No process (not even Apple apps) can access camera or mic without explicit user consent. This is enforced at the kernel level — not just a UI convention.

Privacy Entitlements — every app declares what it needs TCC (Transparency Consent Control) — iOS subsystem that tracks per-app permissions Keychain-backed store — your allow/deny decisions are persisted by iOS
iOS rule: If user previously denied an origin, the dialog is skipped and access is hard-blocked at the OS layer.
L2 WebKit — Browser Sandboxing

Safari uses WebKit. Each website runs in a sandboxed process. The browser mediates between JS and the OS. The JS code cannot directly touch hardware — it asks the browser, which asks iOS, which asks the user.

JS: navigator.mediaDevices.getUserMedia() │ ▼ WebKit Content Process (sandboxed — no direct hardware access) │ ▼ WebKit UI Process (has OS entitlements, shows the system dialog) │ ▼ iOS TCC (checks policy, grants/denies at kernel level)
L3 JavaScript API — The Promise

getUserMedia() returns a Promise. It does not block. The browser shows the permission dialog asynchronously. The Promise resolves with a MediaStream if granted, or rejects with a DOMException if denied.

navigator.mediaDevices .getUserMedia({ video: true, audio: true }) .then(stream => { // ✅ Permission granted // `stream` contains MediaStreamTracks }) .catch(error => { // ❌ Permission denied or error // error.name tells you why: // "NotAllowedError" — user denied // "NotFoundError" — no device found // "NotReadableError" — hardware error });

L5 The Permission Dialog — Out of Your Control

When you call getUserMedia(), iOS shows a system-level alert. The website cannot style it, dismiss it, pre-select Allow, or even detect it. It's rendered by UIKit, not the browser.

1 Your site calls getUserMedia()
↓
2 WebKit asks iOS: "Does this origin have permission?"
↓
3 iOS shows system dialog: "Use your [camera/microphone]?"
↓
4a User taps Allow → Promise resolves → stream delivered
or
4b User taps Don't Allow → Promise rejects → NotAllowedError
L6 MediaStream — What You Actually Get

The resolved Promise gives you a MediaStream. It contains one or more MediaStreamTrack objects. Each track is a live feed of audio or video data that you can render, analyze, mute, or stop.

stream → MediaStream ├── stream.getVideoTracks() → [MediaStreamTrack] │ ├── track.label → "Front Camera" │ ├── track.enabled → true/false (can mute) │ ├── track.readyState → "live" or "ended" │ └── track.getSettings() → { width, height, frameRate } └── stream.getAudioTracks() → [MediaStreamTrack] ├── track.label → "iPhone Microphone" └── track.getSettings() → { sampleRate, channelCount }
To revoke: call track.stop() on every track. The permission is session-scoped. All tracks stopped = permission released.
L7 Lifecycle — Grant, Use, Revoke

Permissions on iOS Safari are ephemeral. They live only for the tab session. Close the tab = gone. But iOS also remembers your last decision per-origin and may skip the dialog next visit.

PERMANENT (OS-level) └ User's Allow/Deny decision → stored by iOS TCC → next visit, iOS may skip the dialog entirely
SESSION (tab-level) └ Active MediaStream tracks → live data flowing → tab close = all tracks killed = stream dies → track.stop() = individual release
REVOKE └ User can revoke: Settings → Safari → Camera/Microphone └ Site cannot detect past grants without trying again └ site.clearAppCache() does NOT clear permissions
Key Rules on iOS Safari

Permissions are per-origin. https://foo.github.io and https://bar.github.io are separate. But http:// and https:// are also separate (HTTP sites get no camera/mic at all on iOS).

Must be user-initiated. On some browsers, getUserMedia() must be called from a user gesture (click/tap). iOS Safari is more lenient but it's best practice.

You cannot style the dialog. It's a UIKit UIAlertController. No CSS, no JS, no customization possible.

You cannot detect the dialog. There is no event, no API, no timing trick. Your Promise just waits.

Stopped tracks = permission released. The OS-level grant may persist for the origin, but the active session ends. Labels go back to masked if you re-enumerate after stopping.

HTTPS required. getUserMedia() is blocked on insecure origins. GitHub Pages provides HTTPS automatically.
