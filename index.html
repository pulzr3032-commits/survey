A guided tour through every layer of the stack — from the physical sensor on your iPhone to the JavaScript API in your browser. Click any layer to explore it in depth.

The Stack — Low to High
Layer 0
Hardware — Sensors & Converters
Physical silicon: image sensors, microphone MEMS, ADCs, ISP
›

What's physically on the device

Every iPhone contains physical components that convert photons and sound pressure waves into electrical signals.

  • Image sensor (CMOS): A grid of photosites. Each photosite converts incoming photons into electrons via the photoelectric effect. The resulting charge is measured and converted to a digital value (RAW pixel data).
  • MEMS Microphone: A microscopic diaphragm that vibrates with sound pressure. Capacitance changes are measured and converted to voltage.
  • ADC (Analog-to-Digital Converter): Converts the microphone's continuous analog voltage into discrete digital samples. On modern iPhones this is 24-bit at up to 48 kHz (or 96 kHz for audio interface input).
  • ISP (Image Signal Processor): A dedicated hardware block in the A-series chip that processes RAW sensor data — demosaicing (Bayer → RGB), noise reduction, tone mapping, white balance, and HDR merging — in real time.
Physical pipeline: Photon → CMOS photosite → Electron charge → ADC → RAW pixel ↓ ISP → YUV420 / NV12 frame Sound wave → MEMS diaphragm → Voltage → ADC → PCM samples (24-bit, 48kHz)

Key insight: At this level, there is no concept of "video" or "audio" — just streams of raw numerical data produced at precise clock rates.

Layer 1
Kernel & IOKit — Driver Layer
XNU kernel drivers, IOKit matching, hardware abstraction
›

How iOS talks to the hardware

iOS uses the XNU kernel (hybrid Mach + BSD). Hardware drivers are loaded as kernel extensions (kexts) managed by IOKit, Apple's object-oriented driver framework.

  • IOKit registry: A tree of driver/service objects. Each hardware component is represented as an IOService.
  • Matching dictionary: When a driver is loaded, IOKit matches it to hardware by comparing vendor/device IDs in the registry.
  • IOMemoryDescriptor: The kernel provides zero-copy memory buffers that drivers write into. User-space frameworks map these buffers directly — no copying needed.
  • Interrupt handling: When the camera sensor finishes a frame or the ADC completes a buffer of samples, a hardware interrupt fires. The kernel driver processes it and signals waiting user-space processes via mach ports.
Kernel-level flow: Hardware interrupt → Driver ISR (Interrupt Service Routine) → IOMemoryDescriptor DMA buffer filled → IOService::message() → notifies clients → User-space framework receives callback → Sample buffer is ready for processing

Key insight: The kernel enforces the first gate of access. Without the driver being loaded and the kernel allowing the process to open the device node, nothing above this layer works.

Layer 2
Core Media / Core Audio — Low-Level C Frameworks
CMTime, CMSampleBuffer, AudioUnit, AudioQueue — the building blocks
›

The fundamental media abstractions

These are C-level frameworks that provide the primitive data types and processing units that everything above builds on.

CoreMedia

  • CMTime: A rational-number timestamp (value/timescale). This is how iOS represents "when" a sample was captured. It avoids floating-point drift.
  • CMSampleBuffer: The universal container for media data. Holds a block of compressed or uncompressed samples (video frames, audio buffers), their CMTime timestamps, format descriptions, and attachments (metadata like rotation, HDR flags).
  • CMBlockBuffer: The actual bytes inside a CMSampleBuffer. Can reference kernel memory without copying.
  • CMVideoFormatDescription: Describes video format — codec type (H.264, HEVC), dimensions, pixel format (NV12, BGRA), color space (sRGB, P3, HLG).
  • CMAudioFormatDescription: Describes audio format — sample rate, bit depth, channel layout, codec (PCM, AAC).

CoreAudio

  • AudioUnit: A modular audio processing node. Think "plugins" — a microphone input unit, an EQ unit, a mixer, an output unit. Chain them together to build a signal graph.
  • AudioQueue: Higher-level API for simple recording/playback. Manages buffers internally.
  • AudioSession: Manages how your app's audio interacts with the system (see Audio Sessions section below).
A CMSampleBuffer contains: ┌─────────────────────────────────┐ │ CMBlockBuffer ← raw bytes │ │ (YUV frame or PCM samples) │ ├─────────────────────────────────┤ │ CMTime ← timestamp │ │ (value: 4800, scale: 600) │ │ = 8.0 seconds │ ├─────────────────────────────────┤ │ FormatDescription ← metadata │ │ (codec, dimensions, etc.) │ ├─────────────────────────────────┤ │ Attachments ← per-sample │ │ (HDR flag, rotation, etc.) │ └─────────────────────────────────┘
Layer 3
AVFoundation — High-Level Objective-C/Swift Framework
AVCaptureSession, AVCaptureDevice, AVAudioRecorder — the app developer's API
›

What native iOS developers use

AVFoundation is Apple's unified media framework that wraps CoreMedia and CoreAudio into an object-oriented, easy-to-use API. This is what Swift and Objective-C developers interact with in native iOS apps.

Core classes

  • AVCaptureSession: The central coordinator. You configure inputs (camera, mic), outputs (video file, sample buffer), and connections between them. It manages the hardware resources and starts/stops the data flow.
  • AVCaptureDevice: Represents a physical device (front camera, rear camera, microphone). You query its capabilities: supported resolutions, frame rates, focus modes, HDR support.
  • AVCaptureDeviceInput: Wraps a device as an input to the session.
  • AVCaptureVideoDataOutput: Delivers raw video frames as CMSampleBuffer objects to a delegate callback. This is where you process each frame in real time.
  • AVCaptureAudioDataOutput: Same but for audio — delivers PCM sample buffers.
  • AVCapturePhotoOutput: High-quality still photo capture with features like RAW, HDR, portrait mode.
  • AVCaptureMovieFileOutput: Records directly to a file with codec selection.
// Native Swift example: let session = AVCaptureSession() let camera = AVCaptureDevice.default(.builtInWideAngleCamera, for: .video, position: .front) let input = try AVCaptureDeviceInput(device: camera!) session.addInput(input) let output = AVCaptureVideoDataOutput() output.setSampleBufferDelegate(self, queue: DispatchQueue(label: "videoQueue")) session.addOutput(output) session.startRunning() // Callback fires ~30 times per second: func captureOutput(_ output: AVCaptureOutput, didOutput sampleBuffer: CMSampleBuffer, from connection: AVCaptureConnection) { // Here's your CMSampleBuffer — raw frame data }

What AVFoundation gives you that the browser doesn't

  • Manual control of focus, exposure, white balance, ISO, shutter speed
  • RAW photo output (ProRAW / DNG)
  • Direct CMSampleBuffer access — process every frame pixel-by-pixel
  • Multi-camera simultaneous capture (Picture-in-Picture, spatial video)
  • Hardware encoder configuration (bitrate, keyframe interval, profile)
  • Background audio recording
  • Custom audio processing graphs with AudioUnit
  • Photo capture pipeline (HDR merge, Deep Fusion, Smart HDR)
Layer 4
TCC — Transparency, Consent & Control
iOS privacy subsystem that enforces permission gates
›

The enforcement mechanism

TCC (Transparency, Consent, and Control) is the macOS/iOS subsystem that intercepts every attempt to access protected resources. It sits between the app and the hardware driver layer.

  • Per-process database: TCC maintains a SQLite database at /private/var/db/com.apple.TCC/TCC.db mapping each process to its allowed/denied/limited status for each resource type (kTCCServiceCamera, kTCCServiceMicrophone, etc.).
  • Interception point: When any process calls a camera/mic API, the system call is intercepted. TCC checks the database. If the status is "not determined," it shows the system dialog. If "denied," it returns an error. If "allowed," the call proceeds.
  • System dialog: Rendered by SpringBoard (the iOS home screen process), NOT by the app. The app has zero control over its appearance, timing beyond the API call, or dismissal.
  • Keychain-backed: User decisions are stored in the Keychain, which means they survive app reinstalls and system updates.
  • Per-origin in Safari: For websites, TCC tracks by origin (scheme + host + port). https://example.com and https://www.example.com are different origins.
TCC decision flow: App calls AVCaptureSession.startRunning() or JS calls navigator.mediaDevices.getUserMedia() │ ▼ TCC intercepts the system call │ ▼ Query TCC.db: "Has this process/origin been granted kTCCServiceCamera?" │ ├─ NOT DETERMINED → Show system dialog │ ├─ User taps "Allow" → Write to TCC.db → Proceed │ └─ User taps "Don't Allow" → Write to TCC.db → Block │ ├─ DENIED → Return error immediately │ (no dialog shown — user must go to Settings to change) │ └─ ALLOWED → Let the call proceed to the driver
Layer 5
WebKit — Browser Sandbox
Process isolation, content process, WebRTC stack
›

How Safari isolates websites

WebKit uses a multi-process architecture. Each website (or group of related sites) runs in a separate sandboxed process that has no direct access to hardware.

  • UI Process: The actual Safari app. Has full OS entitlements, manages windows, tabs, and the permission UI. Talks to the OS directly.
  • Content Process: Runs your website's HTML, CSS, and JavaScript. Sandboxed. No filesystem access, no camera/mic access, no network access beyond what the UI Process proxies for it.
  • Network Process: Handles all network requests on behalf of content processes. Enforces CSP, CORS, cookie policies.
  • GPU Process: Handles WebGL, canvas, and video decoding. Communicates with the actual GPU.

When JavaScript calls getUserMedia(), the content process sends an IPC (inter-process communication) message to the UI Process. The UI Process then talks to TCC. The entire permission flow happens across process boundaries.

WebKit process flow for getUserMedia: Content Process (your JS) │ navigator.mediaDevices.getUserMedia() │ ▼ IPC message to UI Process (Safari app) │ Asks TCC for this origin │ Shows system alert if needed │ ▼ TCC → allows or denies │ ▼ response UI Process → creates AVFoundation capture session internally, gets CMSampleBuffers │ ▼ IPC back Content Process → receives MediaStream object (abstracted, limited data) │ ▼ Your JS gets a Promise with MediaStream

WebRTC stack in WebKit

Behind getUserMedia(), WebKit uses a WebRTC implementation that bridges to the native capture APIs. The browser creates an internal AVCaptureSession, captures frames, and delivers them to the content process as MediaStream objects.

Layer 6
JavaScript API — getUserMedia
MediaDevices, MediaStream, MediaStreamTrack — what your code sees
›

The highest level — what you write

This is the W3C-standard API that web developers use. It abstracts away every layer below into a simple Promise-based interface.

  • navigator.mediaDevices: The entry point. Provides getUserMedia() and enumerateDevices().
  • MediaStream: A container for tracks. Has an id and an active flag. Created by getUserMedia() or HTMLMediaElement.captureStream().
  • MediaStreamTrack: A single audio or video source. Has enabled (mute), readyState ("live"/"ended"), and getSettings() (resolution, frame rate).
  • MediaStreamConstraints: An object specifying what you want: { video: true, audio: true } or more detailed: { video: { facingMode: "user", width: 1920 } }.

The Complete Data Flow
What happens when you tap "Allow" on the permission dialog

This traces a single video frame from photon to your <video> element:

1 Light hits the CMOS sensor. Each photosite generates electrons proportional to photon count.
↓
2 ADC converts analog charge to digital RAW Bayer values. ISP processes to YUV or RGB.
↓
3 Kernel driver DMA-writes frame buffer into shared memory. Signals waiting process via mach port.
↓
4 AVFoundation wraps the buffer in a CMSampleBuffer with CMTime timestamp and format description.
↓
5 WebKit's UI Process captures the frame, decodes if compressed, and sends it via IPC to the content process.
↓
6 Content process creates a MediaStreamTrack. The frame is composited into the <video> element by the rendering engine.
↓
7 Your JavaScript can read the frame via canvas.drawImage(video) or the VideoFrame API (for WebCodecs).

At 30 fps, this entire pipeline runs 30 times per second, end to end. The total latency from photon to screen is typically 50–150 milliseconds on modern iPhones.

Timing & Sample Buffers

CMTime — The Heartbeat of Media

Every sample in iOS media has a timestamp, and those timestamps use CMTime — a rational number (value / timescale) that avoids floating-point drift.

CMTime = value / timescale Example: timescale = 600 Frame 0: value = 0 → time = 0.000s Frame 1: value = 600 → time = 1.000s Frame 2: value = 1200 → time = 2.000s Frame 3: value = 1800 → time = 3.000s Why not float? 1/30 = 0.033333... recurring With floats: tiny errors accumulate over hours With CMTime: 600/600 = exactly 1.0, always

Video: Frames and Frame Rates

Video is a sequence of CMSampleBuffer objects, each containing one frame. The frame rate determines how many arrive per second.

At 30 fps on iPhone front camera (720p): Frame size: 1280 × 720 × 1.5 bytes (NV12) ≈ 1.38 MB Data rate: 1.38 MB × 30 ≈ 41.4 MB/s (uncompressed) With HEVC encoding: Same visual quality ≈ 2–5 MB/s Compression ratio: ~10:1 to 20:1 The browser gives you the DECODED frames. The encoding/decoding happens inside WebKit.

Audio: Samples and Buffers

Audio is a continuous stream of PCM (Pulse-Code Modulation) samples. iOS typically delivers them in buffers of 1024 samples at a time.

At 48,000 Hz (48 kHz), mono, 16-bit: Sample interval: 1/48000 ≈ 0.000021s (21 μs) Buffer of 1024 samples: 1024/48000 ≈ 21.3 ms Bytes per buffer: 1024 × 2 = 2,048 bytes Data rate: 48000 × 2 = 96,000 bytes/s (96 KB/s) The browser delivers audio as a MediaStreamTrack. You can analyze it with Web Audio API's AnalyserNode.
Audio Sessions

What is an Audio Session?

On iOS, an Audio Session is a system-level policy that defines how your app's audio behaves with other system audio. It controls routing (speaker vs earpiece vs Bluetooth), mixing behavior, and interruption handling.

Native iOS audio session categories: AVAudioSessionCategorySoloAmbient → Your audio only. Stops other audio. Muted by silent switch. AVAudioSessionCategoryPlayback → Your audio only. Stops other audio. Plays through silent switch. AVAudioSessionCategoryAmbient → Mix with other audio. Muted by silent switch. AVAudioSessionCategoryPlayAndRecord ← Used for camera/mic → Can play and record simultaneously. → Mixes with other audio (optional). → Default route: earpiece. Must opt-in for speaker. AVAudioSessionCategoryRecord → Record only. No playback. AVAudioSessionCategoryMultiRoute → Multiple input/output routes (USB audio + Bluetooth).
Browser limitation: The browser uses PlayAndRecord internally for getUserMedia({ audio: true }). You cannot change the category from JavaScript. You cannot force speaker output vs earpiece from the browser. Native apps have full control.

Audio Session Interruptions

iOS can interrupt your audio session at any time:

  • Phone call → Your app's audio is paused. Session deactivates.
  • Alarm → Your audio may duck (lower volume) or stop.
  • Screen locks → Background audio may continue if your category allows it.
  • Another app takes focus → Your audio is backgrounded.

In the browser, these interruptions are largely invisible to your JavaScript. The MediaStreamTrack may go silent, the <video> element may pause, but there's no explicit API to detect "audio session interrupted." Native apps receive AVAudioSessionInterruptionNotification.
